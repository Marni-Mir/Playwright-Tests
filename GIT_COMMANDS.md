# Команды Git для работы с изменениями

## 📋 Как запушить в Git только файлы с изменениями

### 1. Посмотреть, какие файлы изменены

```bash
git status
```

Покажет список всех измененных файлов, новых файлов и файлов, готовых к коммиту.

---

### 2. Добавить только конкретные файлы

```bash
# Добавить один файл
git add PW/run-tests.js

# Добавить несколько файлов
git add PW/run-tests.js PW/test-order.json

# Добавить все файлы в папке
git add PW/

# Если в PW/ есть:
# - run-tests.js (изменен, уже отслеживается Git)
# - new-file.js (новый файл, еще не отслеживается)
# - test-order.json (изменен, уже отслеживается Git)

git add PW/          # Добавит все 3 файла
git add -u PW/       # Добавит только run-tests.js и test-order.json (new-file.js не добавится)
```

**Важно:** `git add` добавляет файлы в staging area (область подготовки), но еще не коммитит их.

---

### 3. Закоммитить изменения

```bash
git commit -m "Остановка выполнения тестов при падении первого теста"
```

**Совет:** Пиши понятные сообщения коммита, которые объясняют, что было изменено и зачем.

---

### 4. Запушить в репозиторий

```bash
# Если это первая ветка (первый push)
git push origin <имя-ветки>

# Если ветка уже существует
git push

# Если нужно установить upstream (связь с удаленной веткой)
git push -u origin <имя-ветки>
```

## 🚀 Полный пример для работы с изменениями

```bash
# 1. Посмотреть статус
git status

# 2. Добавить только измененный файл
git add PW/run-tests.js

# 3. Закоммитить
git commit -m "Добавлена остановка выполнения тестов при падении первого теста"

# 4. Запушить
git push
```

## 🔧 Полезные команды

### Посмотреть, что именно изменилось в файле

```bash
git diff PW/run-tests.js
```

Покажет все изменения в файле до добавления в staging area.

### Посмотреть изменения в staged файлах

```bash
git diff --staged
```

Покажет изменения в файлах, которые уже добавлены через `git add`.

### Добавить все измененные файлы (осторожно!)

```bash
git add .
```

Добавит **все** измененные и новые файлы в текущей директории и поддиректориях.

### Добавить только измененные файлы (не новые)

```bash
git add -u
```

Добавит только файлы, которые уже отслеживаются Git и были изменены. Новые файлы не будут добавлены.

### Отменить добавление файла (если добавил по ошибке)

```bash
# Отменить добавление одного файла
git reset HEAD PW/run-tests.js

# Отменить добавление всех файлов (например, после git add -u или git add .)
git reset
# или
git reset HEAD
# или (Git 2.23+)
git restore --staged .
```

Уберет файл(ы) из staging area, но сохранит изменения в файле.

### Отменить все изменения в файле (вернуть к последнему коммиту)

```bash
git checkout -- PW/run-tests.js
```

**⚠️ ВНИМАНИЕ:** Это удалит все незакоммиченные изменения в файле! Используй осторожно.

## 🎯 Если нужно добавить только часть изменений в файле

### Интерактивный режим

```bash
git add -p PW/run-tests.js
```

Это позволит выбрать конкретные изменения (chunks) для коммита. Git покажет каждое изменение и спросит:
- `y` - добавить это изменение
- `n` - пропустить это изменение
- `s` - разбить на более мелкие части
- `q` - выйти

## 📝 Типичные сценарии

### Сценарий 1: Изменил один файл и хочу его запушить

```bash
git status                    # Проверить статус
git add PW/run-tests.js       # Добавить файл
git commit -m "Описание изменений"
git push                      # Запушить
```

### Сценарий 2: Изменил несколько файлов, но хочу запушить только один

```bash
git status                    # Посмотреть все изменения
git add PW/run-tests.js       # Добавить только нужный файл
git commit -m "Описание изменений"
git push
```

### Сценарий 3: Хочу посмотреть изменения перед коммитом

```bash
git diff                      # Посмотреть все изменения
git diff PW/run-tests.js      # Посмотреть изменения в конкретном файле
git add PW/run-tests.js       # Добавить файл
git diff --staged             # Посмотреть, что будет в коммите
git commit -m "Описание"
```

### Сценарий 4: Добавил файл по ошибке, нужно убрать

```bash
git reset HEAD PW/run-tests.js  # Убрать из staging area
# Файл останется измененным, но не будет в коммите
```

### Сценарий 5: Хочу отменить последний коммит (но сохранить изменения)

```bash
git reset --soft HEAD~1
```

Это отменит последний коммит, но оставит все изменения в staging area, чтобы можно было переделать коммит.

## 🔍 Полезные команды для просмотра истории

```bash
# Посмотреть последние коммиты
git log

# Посмотреть последние коммиты в одну строку
git log --oneline

# Посмотреть изменения в последнем коммите
git show

# Посмотреть изменения в конкретном коммите
git show <hash-коммита>
```

## ⚠️ Важные замечания

1. **Всегда проверяй `git status`** перед коммитом, чтобы убедиться, что добавляешь правильные файлы.

2. **Не коммить файлы с секретами** (пароли, API ключи). Используй `.gitignore` для таких файлов.

3. **Пиши понятные сообщения коммитов** - они помогут понять историю изменений.

4. **Не используй `git add .`** без проверки - можно случайно добавить ненужные файлы.

5. **Перед push проверяй**, что коммитишь в правильную ветку.

## 🔄 Предупреждение о окончаниях строк (LF/CRLF)

### Что означает предупреждение?

Если вы видите сообщение:
```
warning: in the working copy of 'file.js', LF will be replaced by CRLF the next time Git touches it
```

**Это не ошибка!** Это информационное предупреждение о том, что:
- **LF** (Line Feed, `\n`) - окончание строки в Unix/Linux/Mac
- **CRLF** (Carriage Return + Line Feed, `\r\n`) - окончание строки в Windows

Git предупреждает, что файл использует LF, но при следующем взаимодействии (checkout, merge и т.п.) он будет автоматически преобразован в CRLF, потому что вы работаете на Windows.

### Что делать?

**Обычно ничего делать не нужно** - это нормальное поведение Git на Windows. Файлы будут работать корректно.

### Если хотите настроить поведение Git:

```bash
# Посмотреть текущую настройку
git config core.autocrlf

# Настроить автоматическое преобразование (рекомендуется для Windows)
git config core.autocrlf true
# При коммите CRLF → LF, при checkout LF → CRLF

# Отключить автоматическое преобразование
git config core.autocrlf false
# Git не будет менять окончания строк

# Сохранять LF в репозитории, но конвертировать в CRLF при checkout (Windows)
git config core.autocrlf input
# При коммите оставляет как есть, при checkout на Windows LF → CRLF
```

**Рекомендация для Windows:** `git config core.autocrlf true`
